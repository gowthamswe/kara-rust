## Clone trait surface for collections

- [ ] **Clone trait surface for collections — vertical implementation (typechecker + interpreter + codegen + runtime).** _(canonical: [phase-8-stdlib-floor.md](phase-8-stdlib-floor.md), search `Clone trait surface for collections`)_

  **P0 / present-day correctness bug.** `design.md § Iteration` line 1692 specs `Vec` / `String` / `Map` / `Set` / `VecDeque` / `SortedSet` / `TreeMap` as `Clone` impls, but every layer is missing today — typechecker has no method registration (the only `"clone"` arm is for `Sender`); interpreter's `eval_method_call` falls through to "method 'clone' not found" panic; codegen has zero dispatch and zero per-type fn emission. Compiled and interpreted binaries calling `Vec.clone()` / `String.clone()` / `Map.clone()` / `Set.clone()` fail today.

  **Consumers blocked** (cross-references): `Set[T] LLVM codegen` subtask 5 (union / intersection / difference) needs per-type clone for non-Copy elements; `Vec.filled[T: Clone]` per design.md §1488; future stdlib generics with `T: Clone` bounds; `#[derive(Clone)]` on user structs whose fields are non-Copy.

  - [ ] **1. Typechecker — `clone` method registration on collection types.** Per-collection method-inference functions register `clone(ref self) -> Self` against `Vec[T]`, `String`, `Map[K, V]`, `Set[T]`, `VecDeque[T]`, `SortedSet[T]`, `TreeMap[K, V]`. The existing `infer_map_method` (`src/typechecker.rs:8275`) gets a `"clone"` arm; `Vec`/`String`/`Set` need their own (today the `Sender.clone()` arm is the only one in the typechecker). The `T: Clone` bound is enforced via the existing trait-bound checking — `Vec[T]: Clone where T: Clone`. Negative test: `Vec.clone()` on `Vec[NonCloneStruct]` rejected with the bound-violation diagnostic.

  - [ ] **2. Interpreter — `clone` dispatch on collection values.** Add a `"clone"` arm to `eval_method_call` (`src/interpreter.rs`) that matches each `Value::Vec` / `Value::String` / `Value::Map` / `Value::Set` / `Value::SortedSet` / `Value::Array` shape and produces a deep copy via Rust-side `Vec::clone` / `String::clone` / etc. The Rust-side clones are correct by construction (each `Value` variant is itself `Clone`); this is purely dispatch wiring.

  - [ ] **3. Codegen — per-type clone fn emission machinery (`emit_clone_fn_for_type_expr`).** TypeExpr-aware emit fn paralleling `emit_hash_fn_for_type_expr` / `emit_eq_fn_for_type_expr` / `emit_display_fn_for_type_expr`. New `clone_fn_cache: HashMap<String, FunctionValue<'ctx>>` keyed on `display_mangle_te(te)`. Function signature is uniform `void karac_clone_<typename>(*const T src, *mut T dst)`. Cases: primitives (load+store), String (call `karac_string_clone`), Vec[T] (recursive elem clone in a loop, malloc fresh buffer), Map[K, V] (iterate via `karac_map_iter_*`, insert into fresh Map with per-K, per-V clones), Set[T] (Map[T, ()]), Tuple (per-field recurse via GEP), `#[derive(Clone)]` struct (field-wise recurse synthesised at the derive site).

  - [ ] **4. Runtime — `karac_string_clone(*const String, *mut String)` helper.** New extern in `runtime/src/string.rs`. Allocates a fresh buffer (same allocator path as `karac_vec_grow`), `memcpy`s `len` bytes from the source, writes `{new_data, len, new_cap}` to the destination. For static literals (source `cap == 0`), allocates a `len`-byte buffer with `cap = len` so the cloned String's scope-exit cleanup correctly frees it (the source's `cap = 0` keeps the static buffer untouched). Declared in `Codegen::new()` alongside the existing `karac_*` externs.

  - [ ] **5. `compile_method_call` dispatch wiring.** Add `"clone"` arms to each per-collection method handler — `compile_vec_method`, `compile_map_method`, `compile_set_method`, plus a String dispatch path. Each wires the source variable's slot through to the corresponding `emit_clone_fn_for_type_expr` and writes the result to a fresh stack alloca + side-table registration so the cloned value participates in scope cleanup correctly.

  - [ ] **6. Scope-cleanup integration.** A cloned `Vec` / `Map` / `Set` / `String` is a fresh heap-owning value — register the appropriate `FreeVecBuffer` / `FreeMapHandle` cleanup action on its scope frame. Goes through the existing `track_vec_var` / `track_map_var` machinery; the only new piece is recognizing that `let w = v.clone()` produces a heap-owning bound name and calling the right tracker.

  - [ ] **7. ASAN tests** in `tests/memory_sanitizer.rs`: `asan_vec_clone_independent_buffers` (source's buffer freed once and the cloned buffer freed once — no aliasing); `asan_string_clone_static_literal` (source has `cap = 0`, clone has `cap > 0` and gets freed, source untouched); `asan_map_clone_independent_handle` (both maps free their bucket arrays separately, plus heap-bearing key/value bytes cloned correctly so neither map points at the other's strings); `asan_set_clone_string_elements` (per-element String clone integrity).

  - [ ] **8. E2E tests** in `tests/codegen.rs` + `tests/interpreter.rs`: `vec_clone_preserves_contents`, `vec_clone_grows_independently` (mutate source after clone, verify clone unchanged), `string_clone_independent_buffer`, `map_clone_preserves_entries` (single-key cases for deterministic output; multi-key behavior verified via cardinality / per-key lookup), `set_clone_preserves_membership`, `derive_clone_struct_with_vec_field` (whole-struct clone produces independent inner Vec).

  - [ ] **9. Empty-collection clone fast path.** `Vec.new().clone()` doesn't need to allocate; same for the other empty collections. Either special-case `len == 0` in the emitted body (skip the malloc; produce a `cap = 0` placeholder) or rely on the runtime's `malloc(0)` defined-behavior path. The former is cleaner and saves an allocation per empty clone.

  **Out of scope for this bullet** (separate items): the fallible-allocation `try_clone` companion methods (covered by `Fallible-allocation API split`); the `#[derive(Clone)]` rejection-when-allocates diagnostic under `panic_on_alloc_failure = false` (also lives in the fallible-alloc bullet).

First stdlib type using `mut ref` on a variant payload — typechecker accepts the borrow-typed fields without changes.
  - [x] **3. `Entry` methods (or_insert / or_insert_with / and_modify)** — new `infer_entry_method` keyed off `Type::Named { name: "Entry", args: [K, V] }`, wired into `infer_method_call`'s named-type dispatch chain. `or_insert(default: V) -> mut ref V` checks the argument against V; `or_insert_with(f: Fn() -> V) -> mut ref V` uses closure-pushdown so `|| Vec.new()` typechecks without an explicit return ascription; `and_modify(f: Fn(mut ref V)) -> Entry[K, V]` returns Entry for chaining.   - [x] **4. Interpreter — `Value::Entry` + dispatch** — `Value::Entry { map_var: Option<String>, key: Box<Value>, slot_idx: Option<usize> }`. The `entry` arm in `eval_method_call` linear-searches `Value::Map`, builds the Entry. `or_insert` / Vacant arm of `or_insert_with` push (key, default) onto the live Map and write back via `env.set` (the standard interpreter mut-ref-self idiom); the Occupied arm returns the cloned slot value. `and_modify` aliases the slot value via `Value::SharedCell` and runs the closure with `[SharedCell]` as the arg so `|v| { v += 1 }` mutates through. `or_insert*` returns the slot value cloned (NOT a true `mut ref V`); the fully-aliased pattern `m.entry(k).or_insert_with(Vec.new).push(row)` is interpreter-incompatible without invasive Map representation changes — codegen handles that case correctly. 7 interpreter tests cover the cardinal cases plus the canonical `and_modify(|v| { v += 1 }).or_insert(1)` chain.
  - [x] **5. Runtime — `karac_map_entry` and `karac_map_lookup_slot`** — two C ABI fns in `runtime/src/map.rs`. The split mirrors the chain-terminal split: `karac_map_entry` (insert-on-vacant) is used by chains terminating in `or_insert` / `or_insert_with` since codegen will write a default through the slot pointer when occupied=false; `karac_map_lookup_slot` (read-only) is used by chains terminating in bare `and_modify` so a missing key never causes an insertion side-effect. Both fns: out-pointer + bool-return shape, resize before probing so the slot pointer is stable for the rest of the call, pointer valid until the next mutating call (`HashMap::entry`'s lifetime contract).
  - [x] **6. Codegen — `m.entry(k)` + chain lowering** — `try_compile_entry_chain` in `compile_method_call` peels `and_modify` wrappers off the receiver and matches `m.entry(k)` at the chain root. One runtime call (`karac_map_entry` or `karac_map_lookup_slot`, terminal-driven) followed by branch blocks for each step, sharing a single slot pointer — exactly one hash per chain. `or_insert(d)` stores the default through the slot on Vacant via standard expression compilation; `or_insert_with(f)` uses the fat-pointer indirect-call path; `and_modify(f)` inlines the closure body directly (binds closure param to a stack alloca initialised from the slot, evaluates the body, stores back through the slot pointer) so `|v| { v += 1 }` mutates the map slot without needing closure-param-as-pointer codegen surgery. Restricted to inline `ExprKind::Closure` — named-fn forms are spec-out-of-scope at v1. Trailing-`and_modify` terminal materialises the Entry struct `{slot_ptr, occupied}` so chained terminals see both halves.
  - [x] **7. Test coverage** — 11 typechecker + 7 interpreter + 7 codegen E2E tests across the work. Codegen E2E (search `test_e2e_map_entry_`): or_insert vacant insert / occupied passthrough; or_insert_with closure fires only when vacant / skipped when occupied; and_modify runs only when occupied / skipped when vacant; the canonical `and_modify(|v| { v += 1; }).or_insert(1)` chain across three calls. The terminal-`.push(row)` form on `or_insert_with(Vec.new).push(row)` is its own piece of work that co-evolves with the per-type Clone codegen bullet (both need a slot-ptr-receiver path on `Vec.push`); the typechecker accepts that chain shape today.
